# Family Calendar App — Implementation Plan

## Context

Build a self-hosted family calendar web app from scratch in an empty Go project. Family members share a single calendar where each person's events are color-coded. The app runs as a single binary with an embedded SQLite database — no external services except an SMTP relay for notifications.

## Stack

| Layer | Choice |
|---|---|
| Backend | Go REST API |
| Frontend | Alpine.js SPA (hash routing) |
| Styling | Tailwind CSS (standalone CLI, no Node.js) |
| Database | SQLite via `modernc.org/sqlite` (pure Go, no CGo) |
| Auth | JWT in httpOnly cookies |
| Email | `gopkg.in/gomail.v2` via SMTP |

## Data Model

```sql
-- migrations/001_initial.sql

CREATE TABLE users (
  id INTEGER PRIMARY KEY,
  email TEXT UNIQUE NOT NULL,
  name TEXT NOT NULL,
  password_hash TEXT NOT NULL,
  color TEXT NOT NULL DEFAULT '#3b82f6',  -- hex
  is_admin INTEGER NOT NULL DEFAULT 0,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE recurrence_rules (
  id INTEGER PRIMARY KEY,
  frequency TEXT NOT NULL,    -- daily | weekly | monthly | yearly
  interval INTEGER NOT NULL DEFAULT 1,
  weekdays INTEGER NOT NULL DEFAULT 0,  -- bitmask: Sun=1,Mon=2,...,Sat=64
  until DATE                  -- null = repeat forever
);

CREATE TABLE events (
  id INTEGER PRIMARY KEY,
  title TEXT NOT NULL,
  description TEXT,
  start_at DATETIME NOT NULL,
  end_at DATETIME NOT NULL,
  all_day INTEGER NOT NULL DEFAULT 0,
  creator_id INTEGER NOT NULL REFERENCES users(id),
  color_override TEXT,        -- null = use creator's color
  recurrence_id INTEGER REFERENCES recurrence_rules(id),
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE event_exceptions (
  id INTEGER PRIMARY KEY,
  event_id INTEGER NOT NULL REFERENCES events(id),
  original_date DATE NOT NULL,  -- which occurrence this replaces
  is_deleted INTEGER NOT NULL DEFAULT 0,
  override_start_at DATETIME,
  override_end_at DATETIME,
  override_title TEXT
);

CREATE TABLE user_notification_prefs (
  user_id INTEGER PRIMARY KEY REFERENCES users(id),
  minutes_before INTEGER NOT NULL DEFAULT 30
);

CREATE TABLE notification_log (
  id INTEGER PRIMARY KEY,
  event_id INTEGER NOT NULL,
  user_id INTEGER NOT NULL,
  scheduled_at DATETIME NOT NULL,
  sent_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

## API Routes

```
POST   /api/auth/login              body: {email, password} → sets httpOnly JWT cookie
POST   /api/auth/logout             clears cookie
GET    /api/auth/me                 → current user

GET    /api/events?from=&to=        → events expanded within window (recurring instances included)
POST   /api/events
GET    /api/events/:id
PUT    /api/events/:id              body includes scope: "this" | "future" | "all"
DELETE /api/events/:id              query param: scope=this|future|all

GET    /api/users                   admin only
POST   /api/users                   admin only — create family member
PUT    /api/users/:id               update name/color/notification prefs
DELETE /api/users/:id               admin only

PUT    /api/users/me/password       body: {current_password, new_password}
```

All `/api/*` routes except `/api/auth/login` require a valid JWT cookie (auth middleware).

On first run with an empty database, the server prints a one-time setup token to stdout. `POST /api/auth/setup` accepts that token to create the initial admin account.

## Key Implementation Details

### Recurrence Expansion

`GET /api/events` expands recurring events in-memory for the requested window. For each base event with a `recurrence_id`, generate all occurrences between `from` and `to`, then merge in `event_exceptions` (skip deleted, substitute overridden fields). Return virtual occurrence objects with a synthetic `id` of the form `{base_id}-{YYYY-MM-DD}`.

### Edit/Delete Scope

- `scope=this` → insert a row in `event_exceptions` (is_deleted or overridden fields)
- `scope=future` → set `recurrence_rules.until` to the day before this occurrence, then create a new base event + rule for the edited version
- `scope=all` → update the base event directly

### Notification Scheduler

Goroutine started in `main.go`, ticks every hour:
1. Query events where `start_at` is within the next 24 hours
2. For each event × each user (all users receive all events), check `user_notification_prefs.minutes_before`
3. Skip if a `notification_log` row exists for `(event_id, user_id, scheduled_at)`
4. Send email via SMTP, insert `notification_log` row

### Frontend Views (Alpine.js, hash routing)

- `#/` — Month grid (CSS Grid 7-col). Day cells show event chips colored by creator color or override.
- `#/event/new?date=YYYY-MM-DD` — Create form: title, date/time, all-day toggle, recurrence picker (none / daily / weekly / monthly / yearly + until date), color override
- `#/event/:id` — Edit/delete form; if recurring, shows scope selector
- `#/settings` — Admin: family member list with color swatches, add/remove members. Personal: change password, notification preference.

## Frontend Style Enhancement — Warm Earth + Purple

### Context

The calendar frontend is functional but visually flat: cold indigo-on-gray palette with minimal depth. Goal: swap to a "warmer and richer" feel using stone/amber as the base and violet as the primary accent. Pure color and shadow token swaps in `cmd/server/web/index.html` — no layout or logic changes.

### Color Palette

| Role | Current | New |
|------|---------|-----|
| Page background | `bg-gray-50` | `bg-stone-100` |
| Card/form background | `bg-white shadow` | `bg-amber-50 border border-amber-200 shadow-md` |
| Primary accent | `indigo-600/700` | `violet-600/700` |
| Accent light bg | `indigo-50` | `violet-50` |
| Today ring | `ring-indigo-400` | `ring-violet-400` |
| Focus rings | `focus:ring-indigo-500` | `focus:ring-violet-500` |
| Links / nav | `text-indigo-600` | `text-violet-600` |
| Text primary/secondary/muted | `gray-900/700/500` | `stone-900/700/500` |
| Borders / dividers | `gray-200` | `stone-200` |
| Calendar nav hover | `hover:bg-gray-200` | `hover:bg-stone-200` |
| Calendar cell gap | `bg-gray-200` | `bg-stone-300` |
| Inputs inside cards | `bg-white` | `bg-white` (unchanged — stands out from amber) |
| Delete actions | `red-*` | unchanged |

### Implementation Steps

1. **Body background**: `bg-gray-50` → `bg-stone-100` on `<body>`
2. **indigo → violet** bulk replace: `indigo-50/400/500/600/700` → `violet-50/400/500/600/700`
3. **gray → stone** bulk replace for neutral text/borders: `text-gray-500/600/700/900`, `border-gray-200`, `hover:bg-gray-200`, `bg-gray-200` (gap wrapper) → stone equivalents (`bg-gray-200` gap → `bg-stone-300`)
4. **Cards**: `bg-white shadow-lg` / `bg-white shadow` → `bg-amber-50 border border-amber-200 shadow-md`; explicit `bg-white` on all `<input>` and `<textarea>` inside cards
5. **Header**: add `bg-stone-50 border-b border-stone-200 shadow-sm` to header wrapper
